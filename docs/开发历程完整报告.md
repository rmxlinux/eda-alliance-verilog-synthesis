# 为 Alliance 添加 Verilog 支持：完整开发历程

## 项目背景

课程大作业的题目是"扩展 Alliance 功能，使其支持 Verilog 语法的逻辑综合"。当我接到这个任务时，首先需要理解 Alliance 是什么，它现在能做什么，以及我们需要补充什么。

### Alliance 工具链架构

Alliance 是一套完整的 VLSI CAD 工具链，由法国 LIP6 实验室开发，包含从行为级描述到门级网表、从逻辑综合到布局布线的全套工具。

**Alliance 的层次结构：**

```
┌─────────────────────────────────────────────────────────┐
│  输入层：VHDL/VBE/FSM → 语言前端解析                     │
├─────────────────────────────────────────────────────────┤
│  中间表示层：VBH/VPN/RTL/BEH                            │
├─────────────────────────────────────────────────────────┤
│  行为综合层：VASY (分析) / BOOM (优化)                  │
├─────────────────────────────────────────────────────────┤
│  逻辑综合层：BOOG (门级映射) / LOON (门级优化)          │
├─────────────────────────────────────────────────────────┤
│  物理层：OCP/NERO (布局布线) / GRAAL/DREAL (查看)      │
├─────────────────────────────────────────────────────────┤
│  输出层：VST/AL 网表 / AP 版图                          │
└─────────────────────────────────────────────────────────┘
```

**关键工具说明：**

- **VASY**：VHDL Analyzer for Synthesis，将 RTL VHDL 解析为内部表示
- **BOOM**：Boolean Minimization，基于 BDD 的布尔表达式优化
- **BOOG**：Binding and Optimizing On Gates，门级映射（技术映射）
- **LOON**：Local Optimization On Nets，门级局部优化
- **MBK**：Memory/Model Base Kernel，Alliance 的核心数据结构库

### 理解 VBE：Alliance 的行为综合边界

在深入实现之前，我需要理解**什么是 VBE，它在 Alliance 中扮演什么角色**。

#### VBE 是什么？

VBE（VHDL Behavioral Entity）是 Alliance 定义的一种**行为级硬件描述格式**，本质上是 VHDL 的一个子集，专门用于表达可综合的行为逻辑。

**VBE 文件示例（简单与门）：**

```vhdl
ENTITY simple_and IS
PORT (
  a : in BIT;
  b : in BIT;
  y : out BIT;
  vdd : in BIT;
  vss : in BIT
);
END simple_and;

ARCHITECTURE behaviour_data_flow OF simple_and IS
BEGIN
  ASSERT (vdd = '1')
    REPORT "power supply is missing"
    SEVERITY WARNING;
  ASSERT (vss = '0')
    REPORT "ground is missing"
    SEVERITY WARNING;
    
  y <= a and b;
END;
```

**VBE 文件示例（寄存器）：**

```vhdl
ENTITY dff IS
PORT (
  clk : in BIT;
  d : in BIT;
  q : out BIT;
  vdd : in BIT;
  vss : in BIT
);
END dff;

ARCHITECTURE behaviour_data_flow OF dff IS
  SIGNAL q_reg : REG_BIT REGISTER;
BEGIN
  ASSERT (vdd = '1') REPORT "power supply is missing" SEVERITY WARNING;
  ASSERT (vss = '0') REPORT "ground is missing" SEVERITY WARNING;
  
  label0 : BLOCK ((clk and not (clk'STABLE)) = '1')
  BEGIN
    q_reg <= GUARDED d;
  END BLOCK label0;
  
  q <= q_reg;
END;
```

#### VBE 在 Alliance 综合链中的角色

VBE 是 Alliance 综合流程的**关键分界点**：

```
输入描述 (VHDL/FSM)
         ↓
      [VASY/SYF]
         ↓
    ┌──────────┐
    │   .vbe   │ ← 行为综合边界
    └──────────┘
         ↓
      [BOOM] ← 布尔优化
         ↓
    .vbe (优化后)
         ↓
      [BOOG] ← 门级映射
         ↓
    .vst/.al (门级网表)
         ↓
      [LOON] ← 门级优化
         ↓
    最终网表
```

**为什么 VBE 如此重要？**

1. **后端的统一输入格式**：BOOM 和 BOOG 都通过 `vhdlloadbefig()` 读取 VBE
2. **语言无关的中间表示**：只要能生成合法 VBE，无论源语言是什么都能进入综合链
3. **明确的语义边界**：VBE 只表达可综合的行为，不包含仿真相关的 delay、四态逻辑细节
4. **标准化的数据结构**：VBE 对应 Alliance 内部的 BEH 数据结构（`befig_list`）

#### VBE 的核心语法元素

**1. 端口声明**

```vhdl
PORT (
  a : in BIT;           -- 输入
  y : out BIT;          -- 输出
  ctrl : inout BIT;     -- 双向
  bus : in BIT_VECTOR(3 downto 0);  -- 向量
  vdd : in BIT;         -- 电源（Alliance 约定）
  vss : in BIT          -- 地（Alliance 约定）
);
```

**2. 信号声明**

```vhdl
SIGNAL tmp : BIT;                    -- 内部信号
SIGNAL data : BIT_VECTOR(7 downto 0); -- 向量信号
SIGNAL q_reg : REG_BIT REGISTER;      -- 寄存器
```

**3. 连续赋值（组合逻辑）**

```vhdl
y <= a and b;                        -- 简单与
z <= (sel and a) or ((not sel) and b); -- 多路选择器
result <= data(3 downto 0);          -- 位选择
```

**4. 寄存器（时序逻辑）**

```vhdl
-- 正边沿触发
label0 : BLOCK ((clk and not (clk'STABLE)) = '1')
BEGIN
  q_reg <= GUARDED d;
END BLOCK label0;

-- 负边沿触发
label1 : BLOCK ((not clk and not (clk'STABLE)) = '1')
BEGIN
  q_reg <= GUARDED d;
END BLOCK label1;
```

**5. 条件赋值（通过 guarded block）**

```vhdl
-- enable 寄存器
label0 : BLOCK ((clk and not (clk'STABLE)) = '1')
BEGIN
  q_reg <= GUARDED ((en and d) or ((not en) and q_reg));
END BLOCK label0;
```

#### 为什么生成 VBE 就能实现 Verilog 集成？

理解了 VBE 的角色后，我意识到：

**关键洞察：VBE 是 Alliance 后端的"API"**

1. **后端工具不关心源语言**
   - BOOM 只关心输入的 `.vbe` 文件格式是否合法
   - BOOG 只关心 BEH 数据结构是否正确
   - 它们完全不知道原始代码是 VHDL 还是 Verilog

2. **VBE 表达能力足够**
   - 可以表达组合逻辑（通过 ABL 表达式）
   - 可以表达时序逻辑（通过 REGISTER + GUARDED BLOCK）
   - 可以表达层次结构（虽然我们选择 flatten）

3. **转换是单向的**
   - 我们只需要 `Verilog → VBE`
   - 不需要 `VBE → Verilog` 反向转换
   - 这大大降低了实现复杂度

4. **复用经过验证的后端**
   - BOOM/BOOG/LOON 已经使用多年，稳定可靠
   - 不需要重新实现逻辑优化、门级映射算法
   - 我们的工作只是"语言翻译"，不涉及综合算法

**因此，我们的策略是：**

```
Verilog 源码
    ↓
[vlog2vbe] ← 我们实现这个工具
    ↓
合法的 .vbe 文件
    ↓
[BOOM/BOOG/LOON] ← 直接复用，无需修改
    ↓
门级网表
```

这就是为什么"生成 VBE"能实现 Verilog 集成的根本原因：**VBE 是 Alliance 后端的标准输入接口，只要我们能生成合法的 VBE，就能无缝接入整个综合链**。

### 任务理解

经过对 Alliance 架构和 VBE 角色的深入理解，我明确了任务边界：

1. **Alliance 已有的能力**：
   - 支持 VHDL/VBE/FSM 作为输入语言
   - 拥有成熟的综合后端：BOOM（布尔最小化）、BOOG（门级映射）、LOON（门级优化）
   - 能够输出 Verilog 网表（通过 VASY 和 MBK 的 `.vlg` driver）

2. **缺失的能力**：
   - **没有 Verilog 作为输入的前端**
   - MBK_IN_LO 不支持 `.vlg` 输入
   - VASY 的 `-v` 选项是 Verilog 输出，不是输入

因此，这个项目的核心不是重写逻辑综合算法，而是**补齐 Verilog 输入前端**，让 Verilog 代码能够进入 Alliance 现有的成熟综合链。

### Verilog 到 VBE 的语义映射挑战

理解了 VBE 之后，我需要思考：**如何将 Verilog 构造映射到 VBE？**

#### 映射表

| Verilog 构造 | VBE 表示 | 挑战 |
|-------------|---------|------|
| `input/output/inout` | `PORT` 声明 | 基本一致 |
| `wire` (组合驱动) | `SIGNAL` | 需要确定信号类型 |
| `reg` (always 赋值) | `SIGNAL` 或 `REGISTER` | 需要区分组合/时序 |
| `assign y = a & b;` | `y <= a and b;` | 表达式转换 |
| `always @*` 组合块 | 连续赋值 | 需要 lowering |
| `always @(posedge clk)` | `BLOCK` + `GUARDED` | 边沿检测转换 |
| `if/else` (组合) | 条件表达式 | 嵌套展开 |
| `case` (组合) | 条件表达式链 | 需要完整覆盖 |
| `[3:0]` 向量 | `BIT_VECTOR(3 downto 0)` | 下标顺序转换 |
| `a[2]` 位选 | `a(2)` | 语法差异 |
| `{a, b}` 拼接 | 位选组合 | 需要逐 bit 展开 |
| `? :` 条件运算符 | `(cond and t) or ((not cond) and f)` | 布尔展开 |

#### 关键语义差异

**1. 向量索引方向**

```verilog
// Verilog: [MSB:LSB]
wire [7:0] data;  // bit 7 是最高位
```

```vhdl
-- VBE: (MSB downto LSB)
SIGNAL data : BIT_VECTOR(7 downto 0);  -- bit 7 是最高位
```

**2. 组合逻辑表达方式**

```verilog
// Verilog: 过程块
always @* begin
  if (sel) y = a;
  else y = b;
end
```

```vhdl
-- VBE: 连续赋值（必须是单一表达式）
y <= (sel and a) or ((not sel) and b);
```

**3. 时序逻辑表达方式**

```verilog
// Verilog: 边沿敏感
always @(posedge clk)
  q <= d;
```

```vhdl
-- VBE: 稳定性检测 + guarded block
SIGNAL q_reg : REG_BIT REGISTER;
label0 : BLOCK ((clk and not (clk'STABLE)) = '1')
BEGIN
  q_reg <= GUARDED d;
END BLOCK label0;
```

**4. 表达式宽度**

Verilog 有复杂的位宽扩展规则，VBE 的 ABL 是 bit-level 的，需要：
- 做宽度推导
- 对向量运算逐 bit 展开
- 处理常量扩展

#### 为什么需要中间 AST？

最初我考虑过"边解析边输出"，但很快意识到不可行：

1. **Verilog `always` 块需要语义分析**：必须先知道整个块的结构，才能判断是组合还是时序，是否有 latch
2. **表达式需要类型和宽度信息**：必须先做宽度推导，才能正确展开向量运算
3. **层次结构需要 elaboration**：实例化、参数传递、flatten 都需要完整的模块信息
4. **错误检查需要上下文**：不支持的构造、信号重复定义等需要在输出前检查

因此，**AST 是必需的中间层**，流程是：

```
Verilog 文本
    ↓
[Parser] 构造 AST
    ↓
[Elaboration] 语义分析、参数展开、层次 flatten
    ↓
[Lowering] 将高级构造降低为基本形式
    ↓
[Emitter] 生成 VBE 文本
```

## 第一步：架构调研与方案选择

### 面临的选择

在开始编码之前，我需要回答一个关键问题：**在哪里接入 Alliance 工具链？**

我考虑了两种方案：

**方案 A：修改 VASY**
- 在 VASY 内部添加 Verilog 输入分支
- 解析 Verilog 后生成 VPN（VASY 的过程网络中间表示）或 RTL
- 复用 VASY 现有的输出 driver

**方案 B：新增独立工具 `vlog2vbe`**
- 创建独立的命令行工具
- 将 Verilog 转换为 Alliance `.vbe` 行为描述
- `.vbe` 是 BOOM/BOOG 已经支持的输入格式

### Alliance 内部数据结构：BEH

在决策之前，我还需要理解 VBE 在 Alliance 内部是如何表示的。

**BEH 数据结构（alliance/src/beh/src/beh.h）**：

```c
typedef struct befig {
  char *NAME;           // 模块名
  struct bepor *BEPOR;  // 端口链表
  struct beout *BEOUT;  // 输出赋值链表
  struct beaux *BEAUX;  // 内部信号赋值链表
  struct bereg *BEREG;  // 寄存器链表
  // ... 其他字段
} befig_list;

typedef struct bepor {
  char *NAME;           // 端口名
  char DIRECTION;       // 'I'/'O'/'B' (input/output/inout)
  char TYPE;            // 'B'/'V'/'M'/'W'/'R' (bit/vector/...)
  // ...
} bepor_list;

typedef struct beaux {
  char *NAME;           // 信号名
  struct chain_list *ABL; // 表达式（ABL 树）
  // ...
} beaux_list;

typedef struct bereg {
  char *NAME;           // 寄存器名
  struct chain_list *ABL; // next-state 表达式
  // ...
} bereg_list;
```

**ABL 表达式树（alliance/src/abl/src/abl.h）**：

```c
typedef struct chain_list {
  void *DATA;
  struct chain_list *NEXT;
} chain_list;

// ABL 表达式是嵌套的 chain_list
// 例如: (AND a b) 表示为 chain_list {DATA=ATOM("AND"), NEXT={a, b}}
```

**两种实现路径**：

1. **直接构造 BEH 数据结构**：
   - 调用 `beh_addbefig()`, `beh_addbepor()` 等 API
   - 构造 ABL 表达式树
   - 调用 `vhdlsavebefig()` 输出

2. **生成 VBE 文本**：
   - 直接输出 VHDL 格式文本
   - 让 BOOM 通过 `vhdlloadbefig()` 读取

### 决策过程

我最终选择了**方案 B（独立工具 + 文本生成）**，原因如下：

#### 为什么选择独立工具而不是修改 VASY？

1. **风险控制**：不破坏 VASY、BOOM、BOOG、LOON 的现有行为
   - VASY 内部的 VBH、VPN、RTL 数据结构复杂
   - 旧代码风格和工具链依赖较重
   - 短时间内很难保证语义正确

2. **快速闭环**：只要能输出合法 `.vbe`，就能立即进入 Alliance 综合链验证
   - 可以立即用 BOOM/BOOG 验证输出正确性
   - 不需要等待完整集成才能测试

3. **独立测试**：`vlog2vbe` 可以单独编译、单独回归，不依赖完整的图形工具环境
   - 不需要 X11/Motif 等图形库
   - 可以在简单的 Linux/WSL 环境中开发
   - 回归测试速度快

4. **渐进演进**：后续如果需要，可以将稳定的 parser 再接入 VASY
   - 第一阶段先验证可行性
   - 第二阶段再考虑深度集成

#### 为什么选择文本生成而不是 BEH API？

这是另一个重要决策。我可以选择：

**方案 A：调用 BEH API**
```c
befig_list *fig = beh_addbefig(NULL, "top");
beh_addbepor(fig, "a", 'I', 'B');
beaux_list *aux = beh_addbeaux(fig, "y");
aux->ABL = createablbinexpr(ABL_AND, 
                             createablatom("a"),
                             createablatom("b"));
vhdlsavebefig(fig);
```

**方案 B：生成 VBE 文本**
```c
fprintf(f, "ENTITY %s IS\n", module_name);
fprintf(f, "PORT (\n");
fprintf(f, "  a : in BIT;\n");
fprintf(f, "  y : out BIT\n");
fprintf(f, ");\n");
fprintf(f, "END %s;\n", module_name);
fprintf(f, "y <= a and b;\n");
```

我选择**方案 B（文本生成）**，原因：

1. **依赖简单**：
   - 不需要链接 BEH/ABL/MUT 等 Alliance 库
   - 不需要理解 Alliance 的内存管理机制（namealloc、链表管理）
   - 可以完全独立编译

2. **调试方便**：
   - 生成的 VBE 文件可以直接查看
   - 可以手工编辑对比，快速定位问题
   - 错误信息更直观

3. **快速迭代**：
   - 不需要深入研究 BEH API 文档（文档较少）
   - 不需要处理 API 版本兼容性
   - 可以快速实现第一版

4. **足够可靠**：
   - VBE 格式相对简单，是标准 VHDL 子集
   - 只要语法正确，BOOM 就能正确解析
   - 可以通过回归测试验证正确性

**未来可以改进**：后续如果需要更强的类型检查和内部一致性，可以将 emitter 替换为 BEH API 构造，但第一版选择文本生成是正确的工程权衡。

### 最终架构决策

这个决策奠定了整个项目的架构：

```
Verilog .v 
    ↓
[vlog2vbe] ← 独立命令行工具
    ↓       - 手写 parser → AST
    ↓       - elaboration (参数、层次)
    ↓       - lowering (always → assign)
    ↓       - emitter (AST → VBE 文本)
    ↓
Alliance .vbe ← 行为综合边界
    ↓
[boom] ← 复用，无修改
    ↓
[boog] ← 复用，无修改
    ↓
[loon] ← 复用，无修改
    ↓
.vst 门级网表
```

**关键洞察**：
- 我们不是在写"Verilog 综合器"
- 我们是在写"Verilog 到 VBE 的翻译器"
- 真正的综合工作由 BOOM/BOOG 完成

### 补充：Verilog 到 VBE 的实例对比

为了更好地理解转换过程，这里给出几个实际的转换例子。

#### 例子 1：简单组合逻辑

**输入 Verilog (simple_and.v)：**
```verilog
module simple_and(input a, input b, output y);
  assign y = a & b;
endmodule
```

**输出 VBE (simple_and.vbe)：**
```vhdl
ENTITY simple_and IS
PORT (
  a : in BIT;
  b : in BIT;
  y : out BIT;
  vdd : in BIT;
  vss : in BIT
);
END simple_and;

ARCHITECTURE behaviour_data_flow OF simple_and IS
BEGIN
  ASSERT (vdd = '1')
    REPORT "power supply is missing"
    SEVERITY WARNING;
  ASSERT (vss = '0')
    REPORT "ground is missing"
    SEVERITY WARNING;
    
  y <= (a and b);
END;
```

**观察点**：
- 自动添加了 `vdd`/`vss` 电源端口
- 添加了 power-supply assertion
- Verilog 的 `&` 转换为 VHDL 的 `and`

#### 例子 2：组合 always 块

**输入 Verilog (always_mux.v)：**
```verilog
module mux2(input sel, input a, input b, output reg y);
  always @* begin
    if (sel)
      y = a;
    else
      y = b;
  end
endmodule
```

**输出 VBE：**
```vhdl
ENTITY mux2 IS
PORT (
  sel : in BIT;
  a : in BIT;
  b : in BIT;
  y : out BIT;
  vdd : in BIT;
  vss : in BIT
);
END mux2;

ARCHITECTURE behaviour_data_flow OF mux2 IS
BEGIN
  ASSERT (vdd = '1') REPORT "power supply is missing" SEVERITY WARNING;
  ASSERT (vss = '0') REPORT "ground is missing" SEVERITY WARNING;
  
  y <= ((sel and a) or ((not sel) and b));
END;
```

**观察点**：
- `always @*` 块被"降低"为连续赋值
- `if/else` 被转换为条件表达式 `(cond and t) or ((not cond) and f)`
- `output reg` 在组合逻辑中仍然输出为普通 `out BIT`

#### 例子 3：时序逻辑

**输入 Verilog (dff.v)：**
```verilog
module dff(input clk, input d, output reg q);
  always @(posedge clk)
    q <= d;
endmodule
```

**输出 VBE：**
```vhdl
ENTITY dff IS
PORT (
  clk : in BIT;
  d : in BIT;
  q : out BIT;
  vdd : in BIT;
  vss : in BIT
);
END dff;

ARCHITECTURE behaviour_data_flow OF dff IS
  SIGNAL q_reg : REG_BIT REGISTER;
BEGIN
  ASSERT (vdd = '1') REPORT "power supply is missing" SEVERITY WARNING;
  ASSERT (vss = '0') REPORT "ground is missing" SEVERITY WARNING;
  
  label0 : BLOCK ((clk and not (clk'STABLE)) = '1')
  BEGIN
    q_reg <= GUARDED d;
  END BLOCK label0;
  
  q <= q_reg;
END;
```

**观察点**：
- 创建内部 `q_reg` 寄存器信号，类型为 `REGISTER`
- 使用 `BLOCK` + `GUARDED` 表达边沿触发
- `clk'STABLE` 是 VHDL 的信号属性，检测信号是否稳定
- `(clk and not (clk'STABLE)) = '1'` 表示上升沿
- 输出端口 `q` 连接到内部寄存器 `q_reg`

#### 例子 4：向量运算

**输入 Verilog：**
```verilog
module vector_ops(
  input [3:0] a,
  input [3:0] b,
  output [3:0] y
);
  assign y = a & b;
endmodule
```

**输出 VBE：**
```vhdl
ENTITY vector_ops IS
PORT (
  a : in BIT_VECTOR(3 downto 0);
  b : in BIT_VECTOR(3 downto 0);
  y : out BIT_VECTOR(3 downto 0);
  vdd : in BIT;
  vss : in BIT
);
END vector_ops;

ARCHITECTURE behaviour_data_flow OF vector_ops IS
BEGIN
  ASSERT (vdd = '1') REPORT "power supply is missing" SEVERITY WARNING;
  ASSERT (vss = '0') REPORT "ground is missing" SEVERITY WARNING;
  
  y(3) <= (a(3) and b(3));
  y(2) <= (a(2) and b(2));
  y(1) <= (a(1) and b(1));
  y(0) <= (a(0) and b(0));
END;
```

**观察点**：
- Verilog `[3:0]` 转换为 VHDL `BIT_VECTOR(3 downto 0)`
- 向量位运算被展开为逐 bit 运算
- 每个 bit 的赋值是独立的语句

通过这些例子可以看到，`vlog2vbe` 的核心工作是**语法转换和语义降低**，而不是逻辑综合本身。

## 阶段 M1：建立最小可运行前端

### 设计思路

第一个里程碑的目标是：**能够解析最简单的 Verilog，生成合法的 .vbe 文件**。

我需要解决三个核心问题：

1. **如何表示 Verilog 代码？** → 设计 AST（抽象语法树）
2. **如何解析 Verilog？** → 实现 parser
3. **如何生成 .vbe？** → 实现 emitter

#### 为什么需要 AST？

我可以选择边解析边输出，但这样做会遇到问题：

- Verilog 的表达式（位选、拼接、条件运算符）和 VBE 的布尔表达式不是一一对应的
- 后续要支持 `always` 块和层次结构，需要中间表示来做转换
- 没有 AST 就无法做语义检查和优化

因此我设计了轻量级的 AST 结构：

```c
VlogModule    // 表示一个模块
VlogSignal    // 表示端口和内部信号
VlogAssign    // 表示连续赋值
VlogExpr      // 表示表达式树
VlogRef       // 表示信号引用、位选、片选
```

#### 为什么使用手写 parser？

虽然 lex/yacc 是标准做法，但第一版我选择了手写递归下降 parser：

- Verilog 子集有限，手写更快落地
- 避免引入额外的生成文件和版本依赖
- 更容易提供清晰的错误信息
- 便于调试和快速迭代

这不是否定 lex/yacc，而是为了课程项目第一版快速闭环做的工程取舍。

### M1 实现内容

第一版支持：

- 单模块文件
- ANSI/non-ANSI 端口声明
- `input/output/inout/wire/reg`
- 标量和向量 `[msb:lsb]`
- `assign` 连续赋值
- 基础表达式：标识符、常量、位选、片选、拼接
- 逻辑运算符：`~ ! & | ^ && || == != ?:`

输出侧实现了基础 VBE emitter：

- Verilog `input/output` → VBE `PORT`
- 内部 `wire/reg` → VBE `SIGNAL`
- `assign` → VBE 并发赋值
- 常量归一化为 `'0'/'1'` 或 `B"..."`
- 向量表达式按需逐 bit 展开

**测试用例**：`simple_and.v`、`mux2.v`、`vector_ops.v`

**验收标准**：`vlog2vbe simple_and.v simple_and` 能生成语法合法的 `.vbe`

**对应提交**：`2fcc931 add verilog initial parsing logic` (2026-06-16 11:52)

## 阶段 M2：支持组合逻辑

### 遇到的问题

M1 只能处理 dataflow 风格的 `assign`，但实际 Verilog 中大量组合逻辑是这样写的：

```verilog
always @* begin
  if (sel) y = b;
  else y = a;
end
```

这种过程块风格的代码需要"降低"（lowering）成连续赋值才能生成 VBE。

### 设计思路

组合 `always @*` 的本质是：**根据输入信号，计算每个输出信号的组合表达式**。

对于 `if/else`：

```verilog
if (sel) y = b;
else y = a;
```

需要 lowering 为：

```
y = sel ? b : a
```

再由 VBE emitter 展开为：

```vhdl
y <= ((sel and b) or ((not sel) and a));
```

对于 `case`：每个 case item 转换为 "selector == label" 的条件，串成条件链。

### 关键决策：latch 处理

组合 always 如果某些路径没有赋值，Verilog 会推断 latch。但：

- Alliance `.vbe` 后端对隐式 latch 支持不如寄存器明确
- 课程项目更关注标准组合/时序综合
- 隐式 latch 容易导致语义不清

因此我**主动拒绝 latch**：

- 如果 `if` 缺少某一路赋值 → 报错
- 如果 `case` 没有 default 或某目标缺少赋值 → 报错

这样避免"看起来能输出，但综合语义不清"的情况。

### M2 实现内容

在 parser 内部引入过程语句结构：

- blocking assignment
- `begin/end` block
- `if/else`
- `case/default`

这些语句不直接输出 VBE，而是先构造临时过程 AST，再降低为连续赋值。

**测试用例**：`always_mux.v`、`always_default.v`、`case_decoder.v`

**对应提交**：`a522ccd add always support` (2026-06-16 12:12)

## 阶段 M3：支持时序逻辑

### 设计目标

让前端从纯组合逻辑扩展到基本时序逻辑，支持：

- `always @(posedge clk)` / `always @(negedge clk)`
- blocking/nonblocking assignment
- enable register
- 常见异步 reset 模板

### 核心挑战：如何表示寄存器？

VBE 使用 `REGISTER` 类型和 guarded block 表达边沿触发行为：

```vhdl
SIGNAL q_reg : REG_BIT REGISTER;
label0 : BLOCK ((clk and not (clk'STABLE)) = '1')
BEGIN
  q_reg <= GUARDED d;
END BLOCK label0;
q <= q_reg;
```

如果 Verilog 输出端口本身是寄存器（`output reg q`），我使用内部存储名 `q_reg`，再把 `q <= q_reg` 接回输出端口。

### AST 扩展

新增 `VlogRegDriver` 用于描述寄存器赋值：

- 目标寄存器
- clock 名称
- posedge/negedge
- guard 条件
- next-state 表达式

### Enable register 的处理

对于：

```verilog
if (en) q <= d;
```

缺失的 else **在时序逻辑里不是 latch**，而是"保持原值"。

因此 M3 和 M2 的策略不同：

- **组合 always**：缺赋值 → 报错
- **时序 always**：缺赋值 → 自动转换为 self reference（`q` 保持）

### 异步 reset 模板

支持常见模板：

```verilog
always @(posedge clk or negedge rst_n) begin
  if (!rst_n) q <= 1'b0;
  else q <= d;
end
```

lowering 时拆成两类 driver：

1. reset 条件下的 level-sensitive guarded assignment
2. clock 边沿下、且 reset 不活跃时的 next-state assignment

这样既保持 reset 优先级，也能被后端按 VBE register 结构处理。

**测试用例**：`seq/dff.v`、`seq/dff_en.v`、`seq/dff_async_reset.v`

**对应提交**：`2bcc308 add always @(posedge clk) support` (2026-06-16 12:35)

## 阶段 M4：支持层次结构

### 为什么需要层次结构？

真实工程中，设计都是模块化的：

```verilog
module and2(input a, input b, output y);
  assign y = a & b;
endmodule

module top(input a, input b, output y);
  and2 u0(.a(a), .b(b), .y(y));
endmodule
```

之前 `vlog2vbe` 只能处理单模块，M4 要支持多模块文件和实例化。

### 核心挑战：elaboration

实例化不只是语法问题。解析到：

```verilog
and2 u0(.a(a), .b(b), .y(y));
```

还需要知道：

- `and2` 模块在哪里？
- `a/b/y` 各是什么方向？
- 子模块端口连接到父模块哪个信号？
- 子模块内部 wire 如何避免命名冲突？
- 子模块里是否还有实例？

因此我新增了 `vlog_elab.[ch]`，将流程拆成：

```
parse design → select top → elaborate/flatten → emit VBE
```

这是 M4 最重要的**结构性变化**。

### AST 扩展

新增：

- `VlogDesign`：表示一个文件中的多个 module
- `VlogInstance`：表示一个 module instance
- `VlogConn`：表示端口连接

parser 从单模块接口扩展为：

```c
vlog_parse_design_file(..., VlogDesign*)
```

### 端口连接支持

支持两种连接方式：

```verilog
and2 u0(.a(a), .b(b), .y(y));  // 命名端口
and2 u1(a, b, y);               // 位置端口
```

要求同一实例内不能混用，这样可以让错误更早暴露。

对于输出端口，要求连接到可赋值 reference，输出端口连接到复杂表达式会报错。

### Flatten 策略

有两个选择：

1. 每个 Verilog module 输出一个 `.vbe`
2. 把层次结构 flatten 成一个 top `.vbe`

我选择 **flatten**，原因：

- BOOM/BOOG 的主路径是单个 `.vbe` 行为描述
- flatten 后不需要处理跨 `.vbe` module 绑定
- 对小规模可综合设计更直观
- 可以复用原来的 emitter

Flatten 核心规则：

- 顶层端口和内部信号保持原名
- 子模块内部信号加实例名前缀：`u_stage_nb`
- 子模块 input reference 替换为父模块连接表达式
- 子模块 output assignment 改写为父模块连接目标
- 子模块内部实例递归展开
- 检测递归实例，避免无限展开

### 后端兼容性修正

联调时发现，旧版 BVL/VBE parser 对标识符规则严格，连续下划线会被拒绝。因此 flatten 命名从双下划线改为单下划线。

还自动追加 `vdd`/`vss` 输入端口，并生成 Alliance 单元库常见的 power-supply assertion，这样生成的 `.vbe` 可以直接进入 BOOG/LOON。

### Top 选择

支持三种方式：

1. 命令行 `-top top`
2. 位置参数 module 名
3. 如果只有一个 module 未被实例化，自动推断它是 top

**测试用例**：`hier/named_two_level.v`、`hier/positional_chain.v`、`hier/nested_modules.v`、`hier/seq_instance.v`

**对应提交**：`df7683b Add multi module support` (2026-06-16 13:22)

## 阶段 M5：参数化支持

### 遇到的新需求

到这里，基础功能已经完整，但真实 Verilog 代码经常使用参数：

```verilog
module pass #(parameter W = 1)(input [W-1:0] a, output [W-1:0] y);
  assign y = a;
endmodule

module top(input [3:0] a, output [3:0] y);
  pass #(.W(4)) u_pass(.a(a), .y(y));
endmodule
```

这涉及：

- 模块参数声明
- 参数在端口范围和信号声明中的使用
- 实例化时的参数覆盖（命名和位置）

### 设计思路

参数化需要在 elaboration 阶段处理：

1. **参数解析**：识别 `parameter` 和 `localparam` 声明
2. **参数求值**：支持常量表达式（加减乘、比较、条件运算符）
3. **参数传递**：实例化时用 `#(...)` 覆盖子模块参数
4. **范围展开**：将 `[W-1:0]` 展开为具体的 `[3:0]`

### 实现挑战

- 参数可以在模块头声明：`module m #(parameter W = 1)(...)`
- 也可以在模块体声明：`parameter W = 1;`
- 实例化支持命名：`#(.W(4))` 和位置：`#(4)` 两种方式
- 参数表达式需要常量求值引擎

### AST 扩展

新增：

- `VlogParam`：表示参数声明
- `VlogParamOvr`：表示实例化时的参数覆盖
- 常量表达式求值函数：`vlog_eval_const_expr()`

在 elaboration 时：

1. 为每个模块建立参数环境
2. 实例化时创建新环境，应用参数覆盖
3. 展开端口范围和信号范围

**测试用例**：`hier/parameterized.v`

**对应提交**：`601b041 add parameter initial support` (2026-06-21 08:22)

## 阶段 M6：Generate 支持

### 为什么需要 generate？

参数化模块通常配合 `generate` 使用：

```verilog
module top #(parameter W = 4)(input [W-1:0] a, 
                              input [W-1:0] b,
                              output [W-1:0] y);
  genvar i;
  generate
    for (i = 0; i < W; i = i + 1) begin : g
      bit_and u_and(.a(a[i]), .b(b[i]), .y(y[i]));
    end
  endgenerate
endmodule
```

这是**静态生成硬件结构**，不是动态循环。

### 设计思路

`generate for` 需要在 elaboration 阶段**静态展开**：

1. 识别 `genvar` 和 `generate for` 语法
2. 根据循环范围展开 N 次
3. 每次展开时：
   - 将 genvar 作为常量绑定到当前迭代值
   - 展开循环体中的连续赋值和实例
   - 为命名块添加迭代索引：`g[0]`, `g[1]`, ...

### 实现挑战

- genvar 只在 generate 块内可见
- 循环体可以包含 assign、instance、嵌套的命名块
- 表达式中可以引用 genvar：`a[i]`, `(i == LAST) ? ~a[i] : a[i]`
- 需要为每次迭代创建独立的信号名空间

### AST 扩展

新增：

- `VlogGenvar`：表示 genvar 声明
- `VlogGenFor`：表示 generate for 循环
- `VlogGenItem`：表示循环体中的语句

Elaboration 扩展：

- `elaborate_generate_for()`：展开循环
- 每次迭代时将 genvar 绑定到参数环境
- 展开后的 assign 和 instance 加入模块

### Generate-if/case 支持

最后一个提交还添加了 `generate if` 和 `generate case` 支持：

```verilog
generate
  if (ENABLE) begin : enabled
    assign y = a;
  end else begin : disabled
    assign y = b;
  end
endgenerate
```

这是**编译时条件选择**，根据参数值静态选择生成哪个分支。

**测试用例**：`hier/generate_for.v`、`hier/generate_if_case.v`、`comb/arithmetic.v`

**对应提交**：
- `3c25344 add generate/parameter support` (2026-06-21 09:26)
- `c6baafe final` (2026-06-21 11:39)

## 关键设计决策回顾

### 1. 不改后端，先补前端

这是项目成功的关键。Alliance 的后端已经围绕 `.vbe` 运转多年，直接复用比重写更可靠。

### 2. 输出 .vbe 而不是直接输出门级网表

如果直接生成 `.vst`，就必须自己做逻辑优化和门级映射。选择 `.vbe` 将项目控制在"语言前端"范围。

### 3. 先手写 VBE emitter，保留未来 BEH API 接入空间

当前手写 VBE emitter 是为了快速闭环。后续可以替换为 BEH API 构造，获得更强的内部一致性。

### 4. 组合 always 禁止隐式 latch

为了保证第一版输出更可控。隐式 latch 虽然可综合，但容易导致语义不一致。

### 5. 时序 always 支持 self-hold

与组合逻辑不同，时序逻辑中的缺失分支表示寄存器保持，符合硬件语义。

### 6. 层次结构采用 flatten

避免了多 `.vbe` 文件绑定问题，对小型设计是最稳妥的方式。

### 7. 静态展开 generate

Generate 在 elaboration 阶段完全展开，emitter 看到的是已经展开后的平坦结构。

2. **选择方案**：独立工具 `vlog2vbe`，输出 `.vbe`，复用后端

```
Verilog .v → vlog2vbe → Alliance .vbe → boom → boog → loon → .vst
```
